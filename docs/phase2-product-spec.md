## 1. Executive Summary

### 배경

Phase I에서 운명의 아이는 AI 작명소, 카드 가챠, 사주 분석, 커뮤니티, 크레딧 결제의 핵심 루프를 구축했다. 그러나 현재 서비스는 **일회성 사용 패턴**에 머물러 있다. 이름을 짓고 나면 다시 방문할 이유가 부족하다.

### Phase II 목표

**매일 돌아오는 서비스**로 전환한다. 구체적으로:

1. **일간 리텐션 드라이버** 확보 -- "오늘의 운수" 콘텐츠
2. **수익화 루프** 강화 -- 소액 반복 과금 모델 (크레딧 소비처 확대)
3. **습관 형성 메커니즘** 도입 -- 출석 체크 + 연속 보상

### 핵심 가치 가설

> IF 등록된 아이의 사주 기반으로 매일 달라지는 운수/육아 콘텐츠를 제공하면,
> THEN 부모 사용자의 주간 리텐션이 40% 이상으로 상승하고 월 ARPU가 2,000원 이상이 될 것이다.
> BECAUSE 부모는 아이에 대한 관심이 일상적이며, "오늘 우리 아이에게 뭐라고 해줄까?"라는 일상 수요가 존재하기 때문이다.

**신뢰도: MEDIUM** -- 사주/운세 앱 시장의 DAU 패턴과 육아 앱의 일간 콘텐츠 소비 패턴에 기반한 가설. 사용자 검증 필요.

## 2. 기능 1: 오늘의 운수

### 2.1 Problem Statement

부모 사용자는 아이 이름을 지은 후 서비스에 재방문할 동기가 없다. 동시에, 부모는 매일 아이에 대한 관심과 걱정이 있으며 "오늘 우리 아이 컨디션은 어떨까?", "뭐라고 칭찬해줄까?" 같은 일상적 궁금증을 갖고 있다. 이 간극을 사주 기반 일간 콘텐츠로 채운다.

### 2.2 Target User (Persona)

| 항목 | 내용 |
|------|------|
| **이름** | 지은맘 (32세, 직장맘) |
| **아이** | 생후 8개월 딸 "지은이" |
| **행동 패턴** | 매일 아침 출근 전 육아 앱 확인, 점심시간에 육아 카페 탐색 |
| **핵심 니즈** | "오늘 우리 아이에게 뭔가 특별한 한마디를 해주고 싶다" |
| **Pain Point** | 육아 정보는 넘치지만, **내 아이에게 맞춤화된** 콘텐츠는 없다 |
| **JTBD** | "나는 매일 아침 **우리 아이만을 위한 맞춤 메시지**를 받아서, 아이와의 시간을 더 의미있게 보내고 싶다" |

### 2.3 Proposed Solution (WHAT)

등록된 아이(children 테이블)의 사주 데이터(saju_data) + 오늘 날짜를 입력으로, Solar Pro 3 LLM이 매일 고유한 6종 콘텐츠 카드를 생성한다.

**콘텐츠 구성 (귀여운 톤앤매너):**

| # | 콘텐츠 | 설명 | 예시 |
|---|--------|------|------|
| 1 | 오늘의 운세 한줄 | 아이 오행 + 일진 기반 | "오늘 지은이는 물의 기운이 강해요. 감성이 풍부한 하루!" |
| 2 | 오늘의 칭찬 한마디 | 사주 성향 기반 칭찬 코칭 | "지은아, 네가 조용히 집중하는 모습이 정말 멋져!" |
| 3 | 오늘의 사랑 표현 | 애착 형성 코칭 | "오늘은 지은이 볼에 뽀뽀 3번 해주세요" |
| 4 | 오늘의 새로운 사실 | 월령 기반 발달 정보 | "8개월 아기는 물건을 양손으로 옮길 수 있어요!" |
| 5 | 오늘의 대화 주제 | 부모-아이 상호작용 가이드 | "오늘은 색깔 이름을 알려주는 놀이를 해보세요" |
| 6 | 오늘의 육아팁 | 사주 성향 맞춤 실용 팁 | "목(木) 기운이 강한 아이에게는 자연 산책이 좋아요" |

**과금 모델:**
- 1회 열람에 운명의 조각 1개 차감 (= 100원 상당)
- 하루에 1번만 새 콘텐츠 생성 (같은 날 재열람은 무료)
- 생성된 콘텐츠는 해당 날짜에 캐싱 (DB 저장)

### 2.4 User Stories

| ID | As a... | I want to... | So that... | Priority |
|----|---------|-------------|------------|----------|
| US-F1-01 | 등록된 아이가 있는 부모 | 메인 화면에서 "오늘의 운수" 카드를 본다 | 오늘 콘텐츠가 있다는 것을 인지한다 | P0 |
| US-F1-02 | 부모 | "오늘의 운수 열기" 버튼을 눌러 조각 1개로 콘텐츠를 연다 | 오늘의 맞춤 콘텐츠 6종을 본다 | P0 |
| US-F1-03 | 부모 | 같은 날 다시 방문하면 추가 비용 없이 콘텐츠를 본다 | 하루 중 여러 번 확인할 수 있다 | P0 |
| US-F1-04 | 부모 | 아이가 여러 명이면 아이별로 각각 운수를 본다 | 각 아이에게 맞춤화된 콘텐츠를 받는다 | P1 |
| US-F1-05 | 부모 | 운수 콘텐츠를 카카오톡/인스타 스토리로 공유한다 | 주변에 서비스를 알린다 | P2 |
| US-F1-06 | 부모 | 지난 날짜의 운수를 히스토리에서 다시 본다 | 과거 콘텐츠를 회상한다 | P2 |

### 2.5 Acceptance Criteria

**AC-F1-01: 콘텐츠 생성 및 열람**
- GIVEN 등록된 아이(saju_data 존재)가 있는 로그인 사용자
- WHEN "오늘의 운수 열기" 버튼을 탭
- THEN 운명의 조각 1개 차감 후 6종 콘텐츠 카드가 순차 표시된다
- AND 조각이 부족하면 충전 안내 바텀시트가 노출된다

**AC-F1-02: 동일 날짜 재열람 무료**
- GIVEN 오늘 이미 운수를 열람한 사용자
- WHEN 다시 "오늘의 운수"에 진입
- THEN 추가 차감 없이 캐싱된 콘텐츠를 표시한다

**AC-F1-03: 일일 콘텐츠 유니크성**
- GIVEN 동일 아이, 동일 날짜
- WHEN 콘텐츠를 요청
- THEN 항상 같은 콘텐츠를 반환한다 (1일 1생성)
- AND 다음 날에는 반드시 다른 콘텐츠가 생성된다

**AC-F1-04: 아이 미등록 시 안내**
- GIVEN 등록된 아이가 없는 사용자
- WHEN "오늘의 운수"에 접근
- THEN "아이를 먼저 등록해주세요" 안내 + 등록 화면 이동 버튼 표시

### 2.6 Scope

#### In Scope (MVP)
- Solar Pro 3 기반 6종 콘텐츠 생성 API
- 일일 캐싱 (daily_fortune 테이블)
- 운명의 조각 1개 차감 로직
- 콘텐츠 카드 UI (스와이프 카드 형태)
- 아이 선택 (복수 자녀 지원)

#### NOT In Scope (명시적 제외)
- 푸시 알림 ("오늘의 운수가 도착했어요") -- Phase III
- 콘텐츠 공유 (카카오톡/인스타) -- Phase III
- 히스토리 캘린더 뷰 -- Phase III
- 운수 기반 커뮤니티 게시 -- Phase III
- 음성 TTS 읽어주기 -- Phase III
- 아이 미등록 상태에서의 "일반 운세" -- 제공하지 않음

## 3. 기능 2: 운명의 조각 (크레딧 리브랜딩)

### 3.1 Problem Statement

현재 크레딧 시스템은 "크레딧"이라는 범용적 이름을 사용하고 있어 서비스 세계관과 괴리가 있다. 또한 크레딧 소비처가 카드 가챠에 한정되어 충전 동기가 약하다. 기존 `credits` 필드와 `destiny_fragments` 필드가 혼재되어 있어 통합 정리가 필요하다.

### 3.2 Target User (Persona)

| 항목 | 내용 |
|------|------|
| **이름** | 민준아빠 (35세, IT 직장인) |
| **행동 패턴** | 가챠/수집 게임에 익숙, 소액 결제에 거부감 낮음 |
| **핵심 니즈** | "무료로 얻을 수 있는 건 얻고, 부족하면 소액 충전하고 싶다" |
| **JTBD** | "나는 매일 조금씩 무료 재화를 모아서, 원하는 기능에 **전략적으로** 사용하고 싶다" |

### 3.3 Proposed Solution (WHAT)

기존 `credits`를 "운명의 조각"으로 리브랜딩하고, 획득/소비 경로를 다양화한다.

**획득 방법:**

| 방법 | 조각 수 | 빈도 |
|------|---------|------|
| 매일 출석 체크 | +1 | 1일 1회 |
| 7일 연속 출석 보너스 | +3 (추가) | 주 1회 |
| 30일 연속 출석 보너스 | +10 (추가) | 월 1회 |
| 이름 추천 공유 | +1 | 공유 시 |
| 이름 투표 참여 | +1 | 투표 시 (일 최대 3회) |
| 유료 구매 | 다양 | 무제한 |

**유료 구매 패키지 (리브랜딩):**

| 패키지 | 조각 수 | 가격 | 개당 단가 | 보너스 |
|--------|---------|------|-----------|--------|
| 작은 보자기 | 10조각 | 1,900원 | 190원 | -- |
| 보통 보자기 | 30조각 (+3) | 4,900원 | 149원 | +3조각 |
| 큰 보자기 | 100조각 (+15) | 12,900원 | 112원 | +15조각, 프레임 1개 |
| 운명의 보자기 | 300조각 (+50) | 29,900원 | 85원 | +50조각, SS급 카드 보장 |

**사용처:**

| 기능 | 소비 조각 | 비고 |
|------|-----------|------|
| 오늘의 운수 열람 | 1조각 | 하루 1회 (재열람 무료) |
| 이름 추천 | 0조각 (3회 무료) → 1조각 | 4회째부터 과금 |
| 카드 뽑기 | 1조각 | 기존과 동일 |
| 상세 작명 리포트 | 5조각 | 기존 1,000원 상당 |

### 3.4 User Stories

| ID | As a... | I want to... | So that... | Priority |
|----|---------|-------------|------------|----------|
| US-F2-01 | 사용자 | UI 전반에서 "크레딧" 대신 "운명의 조각"이라는 표현을 본다 | 서비스 세계관에 몰입한다 | P0 |
| US-F2-02 | 사용자 | 조각 잔액을 항상 상단 바에서 확인한다 | 현재 보유량을 즉시 안다 | P0 |
| US-F2-03 | 사용자 | 조각 충전 페이지에서 리브랜딩된 패키지를 본다 | 세계관에 맞는 패키지를 구매한다 | P0 |
| US-F2-04 | 사용자 | 조각 획득/소비 내역을 타임라인으로 확인한다 | 재화 흐름을 추적한다 | P1 |
| US-F2-05 | 사용자 | 이름 공유/투표로 무료 조각을 획득한다 | 과금 없이도 조각을 모을 수 있다 | P1 |

### 3.5 Acceptance Criteria

**AC-F2-01: 리브랜딩 일관성**
- GIVEN 서비스 전체 UI
- WHEN 사용자가 재화 관련 텍스트를 본다
- THEN "크레딧" 대신 "운명의 조각" 또는 조각 아이콘(퍼즐 피스 모양)이 표시된다
- AND 기존 보유 크레딧은 1:1 비율로 조각으로 자동 전환된다

**AC-F2-02: 무료 획득 상한**
- GIVEN 이름 투표 보상
- WHEN 하루에 3회 이상 투표
- THEN 4회째부터는 조각이 지급되지 않는다 (무한 파밍 방지)

**AC-F2-03: 잔액 부족 시 유도**
- GIVEN 조각 잔액 0인 사용자
- WHEN 유료 기능 사용 시도
- THEN "조각이 부족해요!" 바텀시트 + 출석 체크 안내 + 충전소 바로가기 표시

### 3.6 Scope

#### In Scope (MVP)
- `credits` 컬럼을 `destiny_fragments`로 통합 (DB 마이그레이션)
- UI 전체 "크레딧" → "운명의 조각" 텍스트/아이콘 교체
- 충전 패키지 리브랜딩
- 사용처별 차감 로직 통합
- credit_transactions 테이블의 type에 'attendance', 'share_reward', 'vote_reward' 추가

#### NOT In Scope (명시적 제외)
- 조각 선물하기 (사용자 간 전송) -- 어뷰징 위험, 미구현
- 조각 만료 기한 -- 사용자 불만 우려, 미적용
- 실시간 조각 애니메이션 (획득 시 파티클 효과) -- Phase III
- 조각 환불/현금 전환 -- 법적 검토 필요, 미구현

## 4. 기능 3: 출석 체크 시스템

### 4.1 Problem Statement

서비스에 매일 방문할 인센티브가 없다. 무료 사용자는 크레딧(조각)을 얻을 방법이 유료 구매뿐이어서 이탈률이 높다. 출석 체크는 매일 최소 1회 방문을 유도하는 가장 검증된 습관 형성 메커니즘이다.

### 4.2 Target User (Persona)

| 항목 | 내용 |
|------|------|
| **이름** | 서윤맘 (29세, 전업맘) |
| **행동 패턴** | 매일 아침 스마트폰으로 하루를 시작, 여러 앱에 출석 체크 습관 보유 |
| **핵심 니즈** | "무료로 뭔가 쌓이는 느낌이 좋다, 연속 기록을 깨고 싶지 않다" |
| **JTBD** | "나는 매일 간단한 행동으로 보상을 받아서, **꾸준히 방문하는 보람**을 느끼고 싶다" |

### 4.3 Proposed Solution (WHAT)

앱 방문 시 출석 체크를 통해 운명의 조각을 무료 획득하며, 연속 출석 시 보너스를 제공한다.

**보상 구조:**

| 출석 유형 | 보상 | 조건 |
|-----------|------|------|
| 일반 출석 | +1조각 | 1일 1회 (자정 기준 리셋) |
| 7일 연속 | +3조각 (추가) | 7일 연속 출석 달성 시 |
| 14일 연속 | +5조각 (추가) | 14일 연속 출석 달성 시 |
| 30일 연속 | +10조각 (추가) | 30일 연속 출석 달성 시 |

**월간 최대 무료 획득량 계산:**
- 일일: 30조각
- 7일 보너스: ~4회 = 12조각
- 14일 보너스: ~2회 = 10조각
- 30일 보너스: 1회 = 10조각
- **월 최대 약 52조각** (5,200원 상당)

이는 "오늘의 운수" 30회(30조각) + 카드 뽑기 22회 분량으로, 핵심 기능을 매일 무료로 사용하되 추가 사용 시 과금이 필요한 균형점이다.

### 4.5 Acceptance Criteria

**AC-F3-01: 일일 출석 보상**
- GIVEN 오늘 아직 출석하지 않은 로그인 사용자
- WHEN 출석 체크 버튼을 탭
- THEN 운명의 조각 +1이 즉시 반영되고, 출석 완료 애니메이션이 표시된다
- AND 당일 재출석 시도 시 "이미 출석했어요!" 표시

**AC-F3-02: 연속 출석 보너스**
- GIVEN 7일/14일/30일 연속 출석 달성 사용자
- WHEN 해당 날의 출석 완료
- THEN 일반 보상(+1) + 연속 보너스(+3/+5/+10)가 함께 지급된다
- AND 축하 모달이 표시된다

**AC-F3-03: 스트릭 리셋**
- GIVEN 어제 출석하지 않은 사용자
- WHEN 오늘 출석
- THEN 연속 일수가 1로 리셋된다
- AND "연속 출석이 초기화되었어요. 다시 시작해볼까요?" 메시지 표시

**AC-F3-04: 자정 기준 리셋**
- GIVEN 한국 시간(KST) 기준
- WHEN 자정(00:00)이 지나면
- THEN 새로운 출석이 가능해진다
- AND 전날 미출석 시 연속 일수가 리셋된다

### 4.6 Scope

#### In Scope (MVP)
- attendance_logs 테이블 (user_id, checked_date, streak_count)
- 출석 체크 API (tRPC router)
- 메인 화면 출석 배너 컴포넌트
- 캘린더 UI (이번 달 출석 현황)
- 연속 출석 보너스 지급 로직 (7/14/30일)
- 스트릭 카운트 관리

#### NOT In Scope (명시적 제외)
- 출석 체크 복구권 (빠진 날 메우기) -- Phase III
- 푸시 알림 ("출석 체크 잊지 마세요!") -- Phase III
- 주간/월간 출석 랭킹 -- 경쟁 요소는 의도하지 않음
- 출석 체크 위젯 (iOS/Android) -- 네이티브 앱 전환 시 검토
- 시간대별 차등 보상 (아침 출석 추가 보너스 등) -- 불필요한 복잡도

### 5.1 ARPU 예상

**가정:**
- MAU: 10,000명 (Phase I 런칭 6개월 후 목표)
- 유료 전환율: 5% (업계 평균 3-7%)
- 과금 사용자 월 평균 충전: 12,900원 (프리미엄 팩 기준)

| 지표 | 현재 (Phase I) | 목표 (Phase II, 3개월 후) |
|------|---------------|--------------------------|
| MAU | -- | 10,000 |
| DAU/MAU (Stickiness) | 추정 5% | 25% |
| 유료 전환율 | 추정 2% | 5% |
| 과금 유저 월 ARPU | 추정 4,900원 | 12,900원 |
| 전체 월 ARPU | 추정 98원 | 645원 |
| **월 매출** | -- | **약 6,450,000원** |

**근거:**
- "오늘의 운수"가 일일 방문 동기를 만들어 DAU/MAU 비율을 5배 향상
- 조각 소비처 다양화로 충전 빈도 증가 (월 1회 → 월 2-3회)
- 출석 체크로 무과금 유저도 일일 방문, 전환 기회 증가

### 5.2 리텐션 효과 예상

| D7 리텐션 | 5% | 25% |

## 부록: 기술 참고 사항 (아키텍트 전달용)

> 아래는 Product 관점의 요구사항이며, 기술 설계는 아키텍트가 결정한다.

### 신규 테이블 요구사항

**daily_fortunes:**
- user_id, child_id, fortune_date, content (JSONB, 6종 콘텐츠), fragments_charged, created_at
- UNIQUE(user_id, child_id, fortune_date) -- 1일 1생성 보장
- RLS: 본인만 조회/생성

**attendance_logs:**
- user_id, checked_date (DATE), streak_count, bonus_awarded (JSONB), created_at
- UNIQUE(user_id, checked_date) -- 1일 1출석 보장
- RLS: 본인만 조회/생성

### 기존 테이블 변경 요구사항
- users: `credits` 컬럼 → `destiny_fragments`로 통합 (또는 `credits`를 유지하되 표시명만 변경 -- 기술 판단)
- credit_transactions: type에 'attendance', 'fortune', 'share_reward', 'vote_reward' 추가

### 외부 API 요구사항
- Solar Pro 3 (업스테이지 API) -- 일간 운수 콘텐츠 생성
- 호출 빈도: 최대 DAU x 등록 아이 수 / 일 (캐싱으로 실제는 훨씬 적음)
